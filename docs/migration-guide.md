# Migration Guide

## Purpose

Use this guide when moving a private, local, or older FrameCore-style workflow into a public FrameCore Works Skill Kit-compatible setup.

Migration is not a blind file copy. The goal is to preserve reusable workflow logic while removing anything that is user-specific, provider-specific, private, or unsafe for a public repository.

This guide also defines how maintainers should think about updates from earlier kit versions to later versions.

## Migration Principles

Keep these principles stable across every migration:

- migrate workflow contracts, not private operating context
- use neutral role IDs in source files
- display names stay local and are generated by onboarding
- keep project-local install as the safe default
- keep the provider-neutral boundary intact
- keep `framecore.config.json` local unless a team intentionally shares a sanitized version
- keep `.framecore/manifest.json` as the source of truth for managed install paths
- do not migrate provider credentials, private delivery targets, local absolute paths, generated outputs, backups, or cache folders

If a workflow element cannot be explained as reusable public behavior, do not migrate it into this repo.

## Source Audit

Before moving any content, audit the source workflow manually.

Look for:

- private project names, client context, campaign names, and workspace-specific assumptions
- local paths, cloud folder IDs, emails, credentials, and API-key examples
- provider CLIs, endpoint catalogs, credential setup, and paid execution routes
- generated outputs, working drafts, temp folders, backups, cache folders, and sidecar metadata
- agent display names that identify a private workflow roster instead of a reusable role

Do not migrate provider credentials or any file that exists only to make one private machine or account work.

After the manual audit, run:

```bash
npm run audit:privacy
npm run validate
```

Fix all findings before continuing.

## Role And Naming Migration

Public source files must use role IDs, not personal or private agent names.

Correct migration pattern:

- map each private agent to a neutral role ID from `config/agent-naming.schema.json`
- keep the source template named by role ID, such as `.codex/agents/brief-architect.toml.template`
- keep source examples generic
- let onboarding generate local display names for the target workspace

Do not hard-code local display names into templates, examples, docs, tests, or defaults.

If a role does not exist yet, add it deliberately:

1. Add the role ID to `config/agent-naming.schema.json`.
2. Add a matching `.codex/agents/<role-id>.toml.template`.
3. Add or update handoffs, gates, artifact contracts, docs, and tests.
4. Run `npm run validate`.

## Skill And Agent Migration

Each migrated skill must be operational enough for a new user to understand and apply.

A migrated skill needs:

- clear frontmatter with `name` matching its folder
- `When To Use`, `Inputs`, `Outputs`, `Process`, `Decision Rules`, `Guardrails`, `Handoff`, and `QA Checklist`
- public examples or reusable guidance where needed
- no private project language
- no external paid execution path unless it is documented only as a user-owned extension outside this repo

Each migrated agent template needs:

- role, scope, inputs, outputs, forbidden actions, review gate, and handoff rules
- one clear owner gate
- output expectations that match artifact templates or schemas
- provider-neutral language

## Config And Manifest Migration

Do not treat local installed config as source truth.

`framecore.config.json` belongs to the target workspace. It stores local preferences such as working language, tone, output folder, QA strictness, delivery behavior, and local display names.

`.framecore/manifest.json` belongs to an installed target workspace. It records which files are FrameCore-managed and may include managed hashes. Update, repair, and uninstall use that manifest to avoid touching user-owned files.

For public repo source:

- maintain examples in `config/*.example.json`
- keep schema changes explicit in `config/onboarding.schema.json` and related docs
- keep `schema_version` changes intentional and documented
- do not commit real local install config by default

For an installed workspace:

- review `framecore.config.json` before sharing it with a team
- review `.framecore/manifest.json` before update, repair, or uninstall
- keep backups for manual review until the update is accepted

## Update Repair And Rollback

Use the lifecycle commands by intent:

- `update` refreshes the managed set from the current kit and can add new managed paths.
- `repair` rewrites only paths already listed in `.framecore/manifest.json`.
- `uninstall` removes only exact files listed in `.framecore/manifest.json`.

Before update or repair:

```bash
node scripts/doctor.mjs --mode update --target "$FRAMECORE_TARGET"
node scripts/doctor.mjs --mode repair --target "$FRAMECORE_TARGET"
```

After update or repair:

```bash
node scripts/doctor.mjs --target "$FRAMECORE_TARGET"
npm run validate
```

Rollback is manual and should use the numbered `.bak` files created before managed rewrites. Do not automate rollback by deleting directories. Restore only the file that changed, then rerun doctor/preflight.

## Validation Checklist

Before merging a migrated workflow change:

```bash
npm run cleanup:appledouble -- --apply
npm run audit:privacy
npm run validate
npm test
npm run package:audit
```

For release-oriented migrations, also run:

```bash
npm run release:check
npm run package:list
```

Review package contents before a public tag. The package should contain reusable workflow source, docs, examples, validation, and templates only.

## Release Notes

Every migration that changes install behavior, schemas, agent roles, skill contracts, gates, handoffs, or package contents should update release notes.

Migration notes should state:

- what changed
- whether users need to rerun onboarding
- whether users should run update or repair
- whether any `schema_version` changed
- whether any old config field is deprecated
- whether rollback is a simple backup restore or requires manual reconciliation

Do not tag v1.0 or later releases with undocumented migration behavior.

## Halt Conditions

Stop the migration when any of these are true:

- privacy audit fails
- validation fails
- tests fail
- package audit fails
- source includes private context, local paths, emails, credentials, private cloud IDs, generated outputs, backups, or cache folders
- source includes external paid provider execution paths or credential setup
- docs describe behavior that scripts do not implement
- update, repair, uninstall, or rollback behavior is unclear to a new user
